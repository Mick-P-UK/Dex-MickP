# Purpose

Captured during a claude.ai session on 2026.07.13. Mick noticed that Hermes Agent (Nous Research's open-source agent harness) covers ground that overlaps with several weeks/months of MCSB design work. This note exists purely to preserve the brainstorm for a future decision session against the MCSB PRD (currently v0.2.2). Nothing here has been actioned, built, or decided.

# Reframing: this is a three-way question, not two-way

The instinct is to frame this as "MCSB vs Hermes." It is worth naming that there are actually three systems in the picture, and the real strategic question sits underneath the A/B framing:

1. **PAIDA** - the bespoke harness already partly built for Cedric (Notion Bus 1, GitHub Bus 2, Dex vault as canonical truth, baton-wrap/sundown-wrap session handoff). This is Mick's own hand-built equivalent of what Hermes calls "the harness."
2. **MCSB** - the knowledge/journal layer under design (YAML-driven Obsidian Bases, source/author/generated-by frontmatter split, Supabase pgvector RAG planned for Phase 3b). This is closer to what Hermes calls "memory."
3. **Hermes Agent** - an open-source, model-agnostic harness that already ships both a harness layer AND a layered memory system, plus multi-platform delivery (Telegram, Discord, Slack, WhatsApp, Signal, SMS, email, all from one shared session).

So the real question is not just "do we want Hermes's features" - it is "where do PAIDA, MCSB, and Hermes each sit once we are honest about which of them is the harness and which is the memory." Worth surfacing this before the PRD session so the A/B choice below is not made blind to it.

# Where the overlap actually is

Rough mapping between what MCSB is designing and what Hermes already ships:

- MCSB's source/author/generated-by frontmatter split <-> Hermes's memory.md / user.md distinction (Layer 1 of its memory system)
- MCSB's idea capture via source-type: idea <-> Hermes's automatic USER.md entries written during normal conversation, no explicit "save this" needed
- MCSB's expanded journaling triggers <-> Hermes's periodic review loop (reviews memory after roughly 10 user turns, reviews skills after roughly 15 tool iterations, runs an idle-time pass after a session goes quiet for a couple of days)
- MCSB's Supabase pgvector RAG (Phase 3b) <-> Hermes's Layer 2 full-text session search (FTS-5) over all past conversations, plus pluggable Layer 3 memory providers (Honcho, Memo, Hindsight, Supermemory) for anyone who wants something more sophisticated than the built-in search
- MCSB's Obsidian Bases for YAML-driven views <-> Hermes's dedicated Obsidian skill (Layer 4) - reads, searches, creates, and appends vault notes, using proper wikilinks, scoped to a chosen folder rather than the whole vault
- Nothing in MCSB currently mirrors Hermes's self-created skills (agent notices a repeated workflow and drafts its own reusable skill without being asked) or its single-session multi-platform delivery (same conversation continues across Telegram/Discord/WhatsApp/etc.) - these are the two Hermes capabilities with no real MCSB or PAIDA equivalent yet.

# Option A: Incorporate Hermes (blend with what MCSB has already achieved)

**What this would mean concretely:** Deploy Hermes Agent as the actual harness, point its Layer 4 Obsidian skill at the MCSB vault (scoped folder first), and either use its built-in memory layers as-is or eventually swap in Supabase pgvector as a custom Layer 3 provider once Phase 3b is built.

**In favour:**
- Multi-platform delivery (phone, WhatsApp, Telegram) arrives "for free" instead of being built from scratch
- Self-improving skill creation is a genuinely novel capability neither PAIDA nor MCSB currently has a plan for
- Model-agnostic - Claude stays the reasoning engine underneath, this is not a choice between Hermes and Claude
- Active open-source project shipping fast (multiple releases through Feb-June 2026), so ongoing feature growth is not on Mick's own roadmap to build

**Against / risks:**
- MCSB's careful, bespoke frontmatter schema (ticker removed, exchanges as list, sp_start, view_current values, etc.) was designed deliberately for this specific business. Adopting Hermes's memory conventions wholesale risks flattening that design work rather than building on it.
- Real infrastructure step: an always-on VPS or machine, bot tokens, gateway profiles. Not a toggle - a genuine new piece of standing infrastructure to maintain alongside the vault, Notion, and the GitHub-backed PAIDA architecture already in progress.
- Two "harness" projects running in parallel (PAIDA and Hermes) is a duplication risk unless one is explicitly retired or explicitly demoted to a sub-role.
- Community sentiment on the Obsidian integration specifically warns against enabling too many bundled skills at once, and against giving broad vault access before the pattern is proven - a caution worth taking seriously given the volume of sensitive business and financial data already in Dex-MickP.

# Option B: Do not use Hermes, pinch the good concepts into the MCSB PRD

**What this would mean concretely:** Keep MCSB and PAIDA as the two systems, but pull specific Hermes design patterns into the PRD as first-class requirements rather than emergent behaviour.

**Concepts worth stealing outright:**
- The layered memory model itself - a small, token-budgeted "who is this person" file (Layer 1 equivalent), separate from a full-text searchable archive (Layer 2 equivalent), separate again from a pluggable deeper-memory system (Layer 3 equivalent). MCSB's frontmatter split already gestures at this; Hermes's version is more explicit about the token budget discipline, which MCSB has not yet addressed.
- The review-loop cadence - reviewing memory after N turns, reviewing skills after M tool iterations, and an idle-time nightly pass, rather than relying purely on Mick remembering to trigger a journaling session.
- The "scoped folder first, widen later" discipline for anything writing into Obsidian automatically - a good guardrail MCSB's PRD could adopt explicitly, especially given how much sensitive material sits in Dex-MickP.
- The idea of an agent quietly drafting its own reusable skill after noticing a repeated workflow, as an explicit MCSB Phase 3b/4 feature rather than something borrowed wholesale from Hermes's runtime.

**In favour:**
- No new infrastructure, no duplicated harness, PAIDA stays the single harness
- MCSB's bespoke schema and business-specific design work is preserved rather than replaced
- Selective adoption is lower risk and easier to reverse than a full platform decision

**Against:**
- Loses the multi-platform delivery layer entirely unless PAIDA is separately extended to cover it (which is real, uncosted work)
- Loses the self-created-skills capability, which is the one piece of Hermes with no real substitute anywhere in the current roadmap
- Ongoing manual effort to keep "borrowing ideas" in sync with a fast-moving upstream project, versus just running it

# Option C: Hermes as harness only, MCSB as the brain underneath it (not fully A or B)

Worth naming as a genuine third path rather than a compromise between A and B. Hermes explicitly supports pluggable Layer 3 memory providers and a Layer 4 Obsidian skill - meaning MCSB, once its Supabase pgvector RAG is built in Phase 3b, could in principle become Hermes's actual memory system rather than being replaced by Hermes's built-in one. In this framing:

- Hermes = delivery and harness layer only (multi-platform reach, self-created skills, review-loop scheduling)
- MCSB = the actual knowledge store and RAG layer underneath, doing the same job it was always going to do
- PAIDA's Notion/GitHub two-bus architecture would need a clear role decision here too - either retired in favour of this combined stack, or kept as the Desktop-side counterpart while Hermes handles the "everywhere else" side

This is the option that most avoids throwing away existing work, but it is also the most technically involved of the three, and depends on Phase 3b (not yet built) actually landing before it can be tested.

# Option D: Pilot narrow before deciding anything

Rather than choosing A, B, or C now, run a small, contained Hermes pilot on a single low-risk use case - the idea-capture-via-Telegram case discussed earlier is a good candidate - for a few weeks, on a folder scoped well away from anything sensitive, before touching the MCSB PRD at all. Decide A/B/C afterwards with real evidence of what Hermes actually feels like day to day, rather than deciding from documentation alone.

# Open questions to carry into the PRD fold-in session

- Does PAIDA get retired, demoted, or kept fully separate if Hermes is adopted in any form?
- If Option C is the real target, does that change the design of Supabase pgvector RAG in Phase 3b (e.g. does it need to expose itself in a way Hermes's Layer 3 provider interface can consume)?
- What is the actual infrastructure cost (time, hosting, maintenance) of running Hermes as an always-on service, set against the value of multi-platform reach?
- Is the self-created-skills capability something Mick actually wants running autonomously against business content, or is that specifically a case for human review before anything gets promoted to a real skill?
- Which vault folder, if any, would be the "scoped first folder" for a pilot - needs to be genuinely low-risk given the volume of sensitive material in Dex-MickP.

# Status

No decision taken. No infrastructure created. No PRD changes made. This note is input for a future MCSB planning session.
