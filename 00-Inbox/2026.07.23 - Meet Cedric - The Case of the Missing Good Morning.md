# Meet Cedric - The Case of the Missing "Good Morning"

**Format:** Video (YouTube / Public)
**Project:** Meet Cedric
**Status:** Draft brain-dump (ready to script)
**Hook angle:** A missing greeting exposed three hidden gaps - and how curiosity plus "verify before you act" turned a tiny anomaly into real, permanent fixes.

---

## One-line premise

Cedric forgot to say good morning properly. Instead of shrugging it off, Mick asked "why?" - and that one question unravelled a rule that only half-existed, an auth system that cried wolf, and a backup blind spot nobody had noticed.

## The cold open

Mick starts a normal session. Something is off - Cedric skips his usual time-aware greeting and startup routine. It is a small thing. Most people would ignore it. Mick does not. He asks the question that drives every good debugging session: "Where do you actually get that instruction from, and why did it not fire this time?"

## Beat 1 - The rule that lived in the wrong room

Cedric's greeting rule (good morning / afternoon / evening on London time) existed - but it lived in a file that claude.ai loads and Claude Code does not. Mick's setup has config in several places: user-level (~/.claude), the working folder, and deep inside each vault. Claude Code only auto-reads the first two. The greeting rule was buried in a nested vault folder, so it worked in one environment and silently vanished in another.

The kicker: the same blind spot applied to Cedric's research sub-agent, Ron. Same root cause, waiting to bite.

## Beat 2 - The fix: one canonical copy, loaded everywhere

We promoted the rule to a home every environment reads (~/.claude/_rules.md), kept a single canonical version, and version-stamped it so the two copies can never quietly drift apart. Same treatment for Ron - a byte-identical copy where Claude Code can find him, hash-checked to prove they match. Lesson: in AI systems, WHERE your instructions live decides WHETHER they apply.

## Beat 3 - The happy accident (curiosity pays off)

Cedric warned that fixing a NotebookLM job might need Mick to re-log-in manually. Mick pushed back: "If you can run PowerShell, why can't you just run the login yourself? Try it as an experiment." Cedric did - and discovered the login completes hands-off when the browser session is still alive ("Already logged in.", zero clicks). A recurring "please reauthorise" friction became an automatic self-heal, baked permanently into the pipeline. The whole improvement came from Mick refusing to accept "I can't" at face value.

## Beat 4 - Trust, but verify (the git surprise)

Asked to commit the morning's work, Cedric checked the repository state first instead of blindly running the commands - and found two surprises: Mick had already committed part of it himself between messages, and the code actually lived in a separate, nested repository. Running the "obvious" commands would have errored or done nothing. Lesson: verify the real state before you act; assumptions age badly.

## Beat 5 - The uncomfortable discovery

Chasing the greeting led somewhere bigger: Cedric's core rules and configuration were not backed up anywhere, and a growing automation codebase (ShareScope) was committing only to a local repo with no off-machine copy. A missing "good morning" surfaced two genuine backup gaps - now logged as urgent tasks.

## The takeaway for viewers

- Small anomalies are worth chasing. They are often the visible tip of a systemic gap.
- In AI (and any) systems, the location of your instructions determines whether they run at all.
- "Self-healing" beats "ask the human every time" - automate the recoverable cases.
- Verify before you act. State changes underneath you.
- Curiosity - "just try it" - routinely beats a confident "that will not work".
