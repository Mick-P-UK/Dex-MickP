## Premise / Hook
"The tools we build our lives on change under our feet - quietly, in a release
note most people never read. This month one of mine did. I asked Cedric to
check whether a plan we made in May still held up, and the answer was: mostly
yes, but one important thing had moved. Here is the whole thing explained the
way I actually understood it - with a house, a hallway, a couple of keys and
a front door."

The angle for the audience: you do not need to be a coder to make good
decisions about your own AI setup. You need someone to explain the moving
parts in plain language, and you need to verify claims instead of trusting
them. That is what this episode models.

## The Narrative Arc
1. The check-in. Mick sits down for a couple of hours and asks: where are we
   on the shared-brain project, and is there anything on the to-do list?
   Cedric recaps, and flags one risk item - a two-month-old build plan that
   may have been overtaken by a platform change.
2. The platform change. On 7 July 2026 Anthropic moved Claude Cowork from
   running LOCALLY on the PC to running REMOTELY in the cloud by default.
   Mick had spotted this and suspected it broke the assumption that Cowork
   could read his private credentials file.
3. Verify, do not trust. Rather than take the old plan or even a memory note
   at face value, Cedric tests it live from inside the current session
   (proves the credentials file and the local server are genuinely out of
   reach) AND reads Anthropic's own documentation to confirm why.
4. The subtle correction. Mick's instinct was right that something broke -
   but the thing that broke was not what he feared. His credentials were
   never the weak point for this project; the weak point was HOW Cowork
   reaches the assistant. A networking change, not a secrets change.
5. The plain-English translation. Mick says: I do not know these terms.
   Cedric re-explains the whole thing with one analogy - a house, a hallway,
   two keys and a secure front door - and it clicks.
6. The decision. Mick draws a firm line: no master key ever goes into a
   cloud setting. The admin controls stay where he already works comfortably
   (Claude Code, locally); the cloud tool gets a limited key, for quick note
   capture only.

## The Analogy (the teaching centrepiece - keep this verbatim-ish on camera)
- The assistant (Cedric Server) is a small private office INSIDE your house
  (your PC). It only exists while the house is open.
- "Loopback" just means talking to something in the same house, down your own
  hallway. Nothing goes outside, so it is naturally private.
- You have two keys. A master key (opens everything, including admin
  controls) and a limited key (can only drop a note in the inbox tray).
- A "tunnel" is a secure front door that gives your in-house office one safe
  public address, so a visitor from outside can reach it without you exposing
  the whole house.
- The change: the cloud tool used to live in your house. Now it lives down
  the road. So it can no longer use the hallway - it has to come to the front
  door like a visitor, and we only ever hand a visitor the limited key.
- One sentence: the cloud tool now knocks on the front door with the limited
  key, while the tool that still lives in your house keeps using the hallway
  and the master key.

## Lessons / Takeaways (the teaching value)
- Read the release notes of the tools you depend on. Small changes can move
  big assumptions.
- Verify against the primary source. Cedric confirmed the change from
  Anthropic's own documentation and a live test, not from hearsay or memory.
- Separate the scary word from the real risk. "It cannot read my passwords"
  sounded like the danger; the actual change was about connection, not
  secrets. Naming the difference calmly is half the value of an assistant.
- Keep the powerful key at home. A simple, firm rule - never put the master
  key into a cloud configuration - makes a whole class of decisions easy.
- Plain language is not dumbing down. The same idea, told with a house and a
  hallway, is what let a non-coder make a sound architectural call.

## Why This Matters to the Audience
Most people using AI tools are not engineers. They are being asked to make
real trust-and-security decisions with vocabulary nobody explained to them.
This episode shows that you can stay in charge of your own setup by insisting
on two things: verification, and plain-English explanation. That is a
repeatable habit, not a technical skill.

## ADDENDUM (added end of session 2026-07-10) - the fuller arc

The episode grew a second half over the evening. Two more true-to-life threads
that are strong teaching material:

### Thread B - "Does the working automation still work?"
Mick's real fear: turning up to a webinar and finding the ShareScope demo dead
because it cannot reach his credentials. We checked the actual code rather than
guessing. The reassuring twist: the ShareScope -> Nina (NotebookLM) pipeline was
ALREADY built to run locally on his PC in PowerShell, so the cloud move does not
break it - PROVIDED he runs the demo locally, not from a cloud chat. The visible
browser is a demo positive. The teaching beat: "the scary headline (it cannot read
my passwords) was not the real risk; the real change was quieter - where the thing
runs." Same lesson for the ax-trees plain-language work: it is inherently a
local-execution capability.

### Thread C - the backup that had quietly fallen behind
Mick asked, sensibly, "should we push to GitHub so it is safe in the backup too?"
That instinct uncovered something: his Dex-MickP GitHub backup was 5 commits behind
(the auto-push had silently stalled for about a week), and several of the evening's
files were not in a git repo at all. We could not push from the cloud session (no
credentials there - the same wall as the .env issue), so he ran a short runbook on
his own PC. It was not a clean one-click: a leftover lock file blocked git, then his
own ASCII guard-rail refused a research file with smart quotes in it. We cleared the
lock, set the one problem file aside, and pushed everything else safely.

The teaching value of Thread C is the best of the lot:
- A backup you never test is a backup you cannot trust. His was a week stale and he
  had no idea until he looked.
- The messy middle is normal. A stale lock and a fussy commit hook are exactly the
  small snags real people hit - and they are fixable calmly, one at a time.
- Good guard-rails sometimes say no. His ASCII commit-hook BLOCKED the save. That is
  it working, not failing.
- Verify, do not trust - again. Cedric's own view of a file (via the cloud mount)
  disagreed with the file on Mick's disk, so the fix was deferred to the local tool
  rather than done blind.
