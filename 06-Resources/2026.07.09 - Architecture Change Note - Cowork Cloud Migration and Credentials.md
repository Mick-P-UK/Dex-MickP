# Architecture Change Note - Cowork Cloud Migration and Credentials

## 1. Why this document exists

On 7 July 2026 Anthropic changed how Claude Cowork runs: sessions now execute REMOTELY on
Anthropic's servers by default, where before Cowork ran locally on the PC. This is not just a
cosmetic change (the merge of the Chat and Cowork tabs into a single "Home" tab, with "Code"
kept separate). It changes what a session can and cannot reach on Mick's computer, and in
particular it broke the assumption that Cedric can read the credentials file at
C:\Users\pavey\.env.

This note is the permanent reference for that change. It records what changed, what the
Anthropic documentation actually says about credentials and file access, why one of our
routines broke, which OTHER pieces of prior work may be affected, and the options for
handling credentials going forward. Link to this note from any workflow that depends on
local files or secrets.

## 2. Summary (the short version)

- Cowork sessions are now remote by default. The agent loop and code both run in an
  isolated sandbox on Anthropic servers.
- A remote session can reach the PC ONLY through the Claude Desktop app, ONLY for folders
  Mick has explicitly connected, and ONLY while the app is open. Anything outside a
  connected folder - including C:\Users\pavey\.env at the home-folder root - is unreachable.
- Anthropic does NOT provide a way to store a .env or any long-lived secret inside a cloud
  Cowork sandbox. The sandbox holds only short-lived, session-scoped tokens that expire
  within hours. Long-lived service credentials are meant to live behind CONNECTORS, where the
  authorization token stays server-side and never enters the sandbox.
- Local execution still exists, but only through the Claude Desktop app's older model, and it
  still runs code in an isolated VM. For full native local file access (reading .env
  directly), the reliable tool is Claude Code running locally on the PC.
- Net effect for us: any workflow that expected to read C:\Users\pavey\.env or reach files
  outside a connected folder must either (a) run in Claude Code locally, or (b) be re-plumbed
  to use a connector / server-side credential store.

## 3. What changed, with dates and sources

| Date | Change | Source |
|------|--------|--------|
| 12 Jan 2026 | Cowork launched: runs locally on the PC in an isolated VM, with direct access to local files and MCP integrations. | Release notes (Jan 2026) |
| 9 Apr 2026 | Cowork generally available on macOS and Windows (still local). | Release notes |
| 7 Jul 2026 | Cowork on web and mobile; sessions now run REMOTELY by default, saved to the Claude account; Chat and Cowork share one "Home". | Release notes; Cowork on web/desktop/mobile |

The 7 July release is the pivotal one. The tab rename is the visible part; the move to
remote-by-default execution is the part that changed file and credential access.

## 4. How the two architectures differ

| Property | Local session (old default) | Remote session (new default) |
|----------|-----------------------------|------------------------------|
| Where code runs | On the PC, in an isolated VM (Hyper-V on Windows) | In a temporary sandbox on Anthropic servers |
| Local file access | Connected folders, via the on-device agent | Only via the Desktop app, only connected folders, only while the app is open |
| Files outside connected folders (e.g. .env at home root) | Reachable by the on-device agent in the older model | NOT reachable |
| Long-lived secrets in the runtime | N/A | NOT supported; sandbox holds only short-lived session tokens (expire in hours) |
| Connector auth tokens | N/A | Never enter the sandbox; connector calls run server-side |
| Network | Egress filtered per policy | No network by default; egress via mandatory allow-listed proxy |
| Works with PC off | No | Yes for cloud-only work; NO for anything that needs local files or the browser |
| Audit logging | Not in Compliance API / audit logs (OpenTelemetry only, Team/Enterprise) | Same |

## 5. What the documentation says about credentials and .env (the direct answer)

Mick's question was whether Anthropic documents any way to store .env files in the cloud.
The answer from the primary sources is:

1. There is NO documented mechanism to store a .env file, or any long-lived secret, inside a
   cloud Cowork sandbox. The architecture overview states plainly: "Short-lived credentials
   only. The sandbox holds only session-scoped tokens that expire within hours."

2. The intended secure pattern for giving Claude authenticated access to an external service
   is a CONNECTOR: "Connector authorization tokens never enter the sandbox; connector calls
   are made on the server side." So the credential lives in Anthropic's server-side connector
   layer (OAuth), not in a file the sandbox reads.

3. Anthropic explicitly advises AGAINST exposing credential files to Cowork's local file
   access. The safety guidance says: "be cautious about granting access to sensitive
   information like credentials, or personal records" and "Avoid granting access to local
   files with sensitive information." It recommends a dedicated, non-sensitive working folder.

4. Cowork activity is NOT captured in audit logs or the Compliance API today (only via
   OpenTelemetry on Team/Enterprise). Worth knowing for anything credential-touching.

Conclusion: "put the .env in the cloud" is neither supported nor advised. The supported
choices are: keep secrets LOCAL and run the job locally (Claude Code), or move the secret
behind a CONNECTOR / server-side credential store so the token never sits in a readable file.

## 6. Why our end-of-month portfolio routine broke (worked example)

The EOM portfolio posting SOP assumed "Claude Desktop with Filesystem MCP can read
C:\Users\pavey\.env". Under the new remote model that is false: the sandbox could reach
WordPress and Yahoo over the network, and (once connected) could read the DIY - Portfolios
folder, but it could not read .env, because .env sits at the home-folder root, which Cowork
will not let us connect. We resolved it for June by generating a local Python script that
reads .env itself at run time, and going forward by running the routine in Claude Code
locally. The SOP has been corrected to v1.1 accordingly.

## 7. Repercussions on other prior work (register to review)

The same failure mode applies to anything built under the old local-Cowork assumptions that
reads C:\Users\pavey\.env, reads a home-directory file, or reaches files outside a connected
folder. This register is Cedric's assessment and should be verified item by item, not taken
as final.

| Workflow / skill | Depends on | Risk under cloud Cowork | Fix |
|------------------|-----------|-------------------------|-----|
| WordPress publishing (portfolio-post-creator, wordpress-image-uploader, wordpress-post-publisher) | C:\Users\pavey\.env | HIGH - confirmed broken | Claude Code local, or a WordPress connector/MCP |
| ShareScope / Stockopedia automation (ax-trees-automation) | C:\Users\pavey\.env for platform logins | HIGH | Claude Code local; browser steps need Desktop app open |
| NotebookLM CLI workflows (notebooklm-*, sharescope-nlm-*) | Auth at C:\Users\pavey\.notebooklm (home dir) | HIGH - home dir not connectable | Claude Code local |
| Xero pipelines (natwest/cc1136/ii/schwab/paypal-to-xero) | Local bank files and output folders (not secrets) | MEDIUM - only if their folders are not connected | Connect the specific folders, or run in Claude Code |
| yt-weekly-stats-v2 | Claude in Chrome + local files | MEDIUM - browser works only via Desktop app while open | Run with Desktop app open, or Claude Code |
| gmail-self-notes, cedric-note-fetcher | Gmail + Google Drive CONNECTORS | LOW - connectors run server-side, the blessed path | No change expected |
| MCSB / Cedric Server | Its own server env, separate from Cowork | LOW - separate runtime | Review its own assumptions independently |

Action: walk this register at the next convenient point and confirm each item. The pattern to
search for in any skill is a hardcoded read of C:\Users\pavey\.env or any C:\Users\pavey\...
home path, or an assumption of broad local file access.

## 8. Options for credentials going forward

| Option | How it works | Pros | Cons |
|--------|-------------|------|------|
| A. Claude Code local (recommended default) | Run the job in Claude Code on the PC; it uses Mick's OS permissions and reads .env directly. | No credential move; single-source .env rule intact; reuses existing skills. | PC must be on; not fully unattended without OS scheduler. |
| B. Connector / MCP (Anthropic's documented secure path) | Access the service through a connector; the OAuth token stays server-side and never enters the sandbox. | Works from cloud, PC can be off; matches Anthropic's intended model; no secret in a file. | No managed connector exists for a self-hosted WordPress (Hostinger); would need a custom MCP. A WordPress.com connector exists but is for WordPress.com-hosted sites, not diy-investors.com. |
| C. n8n encrypted credential store | Keep secrets in n8n's encrypted credentials; n8n runs the pipeline (or Claude calls n8n via its connector / a webhook). | Fully server-side, PC off; n8n is already in the stack; a proper secret store, not a plaintext file. | Bigger build; relaxes the "creds only in .env" rule; AI content steps still need a Claude API call. |
| D. Plaintext .env in the cloud | (Not available) | - | Not supported by Anthropic and explicitly advised against. Do not pursue. |

Recommended stance: use A (Claude Code local) as the default for any .env-dependent job now,
and treat B or C as the route only if and when "PC switched off" automation becomes a real
requirement. If we go to B/C we must consciously decide to move the specific secret out of
.env into a connector or n8n's store, and record that decision.

## 9. Open questions to watch

- Will Anthropic add a first-class secrets mechanism for cloud Cowork sandboxes? None exists
  today. Re-check the release notes periodically.
- Does a self-hosted WordPress connector / MCP become available (removing the need for local
  .env for publishing)?
- For any workflow moved to a connector or n8n, confirm the audit-logging gap (Cowork is not
  in the Compliance API) is acceptable for that data.
- Confirm each item in the section 7 register with a live check before relying on it.
