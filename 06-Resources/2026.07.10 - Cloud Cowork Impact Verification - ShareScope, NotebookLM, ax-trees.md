# Cloud Cowork Impact Verification - ShareScope, NotebookLM (Nina), ax-trees

This walks the section-7 register of the 9 July Architecture Change Note with a
LIVE code inspection, not an assumption. It answers Mick's question: did the
7 July cloud-Cowork move break work that was previously working?

## Method

- Inspected the actual scripts and skills (not just the register).
- Confirmed from THIS cloud session that C:\Users\pavey\.env is unreadable and
  localhost/local browser are unreachable (sandbox is a remote VM at 172.16.10.3).

## Headline answer

NONE of the code broke. The scripts are unchanged and still work. What changed
is WHERE they must be run. Everything here is LOCAL-execution work: it reads
C:\Users\pavey\.env and drives a real browser on Mick's PC. A cloud Cowork
session cannot do either. Run these LOCALLY (Claude Code on the PC, or the
existing PowerShell / .bat launchers) and they behave exactly as before. Try to
run or drive them FROM a cloud Cowork chat and they fail.

## Area 1 - ShareScope automation -> NotebookLM (Nina)

Finding: this pipeline was ALREADY designed to run locally. The canonical skill
(sharescope-nlm-upload) states in its own architecture note:
  "Both scripts use headed Windows Playwright and the notebooklm CLI with Mick's
   local auth, so they run in PowerShell on Mick's machine. Cedric cannot execute
   them from the sandbox."
So the pipeline sits on the SAFE side of the migration. Cedric's role was always
pre-flight checks and verification; Mick runs the actual commands locally.

- Credentials: sharescope_login.py and sharescope-login.js read
  C:\Users\pavey\.env (SHARESCOPE_USERNAME / SHARESCOPE_PASSWORD) via dotenv, in
  the LOCAL Python/Node process. Unaffected when run locally.
- Browser: headed Playwright = a visible browser on Mick's PC. Good for a
  webinar; impossible from the cloud sandbox.
- NotebookLM CLI (Nina): notebooklm-py with Mick's local auth
  (C:\Users\pavey\.notebooklm). Runs locally.

Verdict: NOT broken for a webinar demo, PROVIDED it is run locally. The visible
browser is a demo positive. Do NOT attempt the live demo from a cloud Cowork
session.

INCONSISTENCY TO FIX (important): the vault master rule

## Area 2 - ax-trees-automation (plain-language / AI-UI access to ShareScope)

Finding: same shape, but MORE exposed to the cloud shift than the scripted
pipeline, because the ambition is Claude driving ShareScope LIVE through the
browser (Playwright accessibility trees) in plain language, in real time.

- Credentials: sharescope-login.js reads C:/Users/pavey/.env via dotenv and
  saves auth to ax-trees-automation/auth/sharescope-auth.json. Local only.
- The vision (agent + sub-agents reading/operating ShareScope via the UI) relies
  on a LIVE local browser plus a local Playwright/automation driver controlled by
  the agent as it reasons. Per Anthropic's architecture doc, "local MCP servers
  don't run in remote sessions" and a remote session reaches the device only
  through the Desktop app for connected folders - NOT a live local browser it can
  puppet. So driving ShareScope live from a cloud Cowork chat will not work.

Verdict: the built scripts run fine locally. The live plain-language driving of
ShareScope must run from a LOCAL execution surface (Claude Code on the PC; or, if
still enabled for the desktop deployment, a local Cowork session). It cannot be
done from a cloud Cowork chat. This is the item to keep clearly in mind before
we invest more in the AI-UI direction: it is inherently a local-execution
capability.
