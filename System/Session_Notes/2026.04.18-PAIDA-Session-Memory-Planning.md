## What Triggered This Discussion

During the /daily briefing session (2026.04.18), Mick observed that session logs are
not currently being stored permanently or reliably. The question arose: how do we
create a permanent, searchable memory of every PAIDA session so that nothing is ever
truly lost?

---

## Problem Statement

The current session_log.md is a single file that has been overwritten rather than
appended consistently. Older sessions are not reliably archived. There is no
searchable layer. Short ad-hoc notes have no home. The Session_Notes/ folder exists
but is empty. Session_Learnings/ has some dated files from Jan-Mar 2026 but stopped
being populated in March.

**The gap:** No permanent, cumulative, searchable record of PAIDA sessions exists.

---

## Requirements (agreed in discussion)

1. Every session logged - regardless of length or whether "significant" work was done
2. Short notes must be droppable - quick ad-hoc observations, decisions, reminders
   that aren't a full session but need to be captured
3. Retrieval by: phrase, concept, document name, idea, company name
4. Permanent - nothing deleted, archive grows over time
5. Practical - must be maintainable without heavy overhead each session

---

## Architecture Options Discussed

### Option A: Vault only (Git-backed flat files)
- Permanent and version-controlled via GitHub
- Readable by Filesystem MCP in Claude Desktop
- BUT: requires manual commits, local machine dependency, not semantically searchable

### Option B: Notion (cloud database)
- Always online, searchable, accessible anywhere
- Notion MCP allows me to write to it from Claude Desktop
- BUT: requires MCP connection (not always available), no RAG query capability

### Option C: Dual - vault + Notion
- Most robust but highest maintenance overhead

### Option D: NotebookLM as query layer (Mick's suggestion - PREFERRED DIRECTION)
NotebookLM has a light RAG system. Sources added to a notebook are indexed and
queryable by phrase, concept, company name etc. - exactly matching the retrieval
requirements above.

**Key insight from Mick (2026.04.18):** NotebookLM Studio has a "note" feature
where you can type or dictate short notes directly. These notes persist in the
notebook and can be promoted to sources. This solves the "drop a short note in"
requirement neatly.

---

## Emerging Architecture (not yet finalised)

### Layer 1: Vault - permanent raw archive
- Individual dated session files in a new Session_Archive/ folder
- Format: YYYY.MM.DD_session.md
- Never overwritten, ever - append only to the archive
- Git-backed via commit_session.bat
- Also serves as the source material for NotebookLM uploads

### Layer 2: NotebookLM - "PAIDA Session Memory" notebook (new, 4th notebook)
- This would be a 4th NLM notebook alongside the 3 planned but not yet created:
  1. Inner Circle Webinars (planned, not created)
  2. Plaza Group Webinars (planned, not created)
  3. AI for Investing Webinars (planned, not created)
  4. PAIDA Session Memory (NEW - proposed)

**How it would work:**
- NotebookLM Studio note = home for short ad-hoc notes (typed or dictated)
- Weekly: Studio note promoted to a source (or compiled session files added as source)
- Periodic: Compiled session logs added as dated source documents
- Query: Ask the notebook "when did we last work on X?" or "what did we decide about Y?"
  and get cited answers back

### Cadence options (NOT YET DECIDED - needs Mick input):
- **Weekly sources:** Fresh, nothing more than 7 days old is un-queryable. More overhead.
- **Monthly sources:** Slightly stale but manageable. One upload task per month.
- Mick was undecided on this point when he left - needs to be settled on return.
